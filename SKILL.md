# seo-text — собрать данные, написать текст, проверить

Этот скил — второй фронтенд к движку desktop-приложения seo-bundle: те же чистые
алгоритмы из `src/lib`, но без Electron. Ты сам выступаешь той нейросетью, в
которую обычно вставляют промпт, — то есть пишешь финальный текст и проверяешь
его. Работай строго по шагам.

## Вход (спроси у пользователя, чего не хватает)

- **Serpstat API-токен** (обязательно; берёт ключи и конкурентов). Без него —
  ручной режим `--no-serp` с `--kw "a,b,c"` и `--url "https://..,.."`.
- **Категория / главный ВЧ-ключ** (обязательно) — `--category`.
- **Сайт пользователя** — `--site shop.ua` (исключается из конкурентов).
- Необязательно: тип страницы `--type` (category|product|brand|filter |
  guide|explainer|listicle|comparison | content|catalog|landing|eduguide;
  по умолчанию `category`), язык `--lang` (ru|uk|en, по умолч. ru),
  регион `--region` (ua|ru|us|…), факты о магазине/сервисе/авторе `--facts "…"`,
  число конкурентов `--count` (по умолч. 10).
- **Доп. пожелания заказчика** (свободный текст после стандартных параметров) —
  `--extra "…"`: «упомянуть бренд X», «дать ссылку на картинку <url>», особый блок
  и т.п. Уйдёт в промпт блоком ДОПОЛНИТЕЛЬНЫЕ ТРЕБОВАНИЯ. Разбери просьбу:
  - требование к **объёму** («минимум 4000 знаков») → передай также `--volume-zbp 4000`
    (для EN — `--words 700`), чтобы и промпт, и проверка ориентировались на эту цифру;
  - микроразметка → `--schemas product,breadcrumb`;
  - **картинку/ссылку** ты как автор вставляешь, даже если тег вне базового
    whitelist (img/a) — явное пожелание заказчика приоритетнее, добавь аккуратно.

Предупреди один раз: запуск тратит кредиты Serpstat (несколько вызовов).

## Шаг 1. Собрать данные и инструкцию

Запусти из корня репозитория, сохранив вывод в файл:

```bash
node ~/.claude/skills/seo-text/collect.js \
  --token "$SERPSTAT" --category "<ВЧ-ключ>" --site "<сайт>" \
  --type category --lang ru --region ua [--facts "<факты>"] \
  > ~/.claude/skills/seo-text/run.json
```

Прочитай `run.json`. Если `ok:false` — покажи `error` и остановись. Иначе:
- `prompt` — **полный набор инструкций нужного документа и языка**. Это твой ТЗ.
- `brief` — параметры для проверки (главный ключ, ключи, LSI, объём, тип, язык).
- `faqSeeds` — реальные вопросные запросы (PAA) для блока FAQ.
- `diagnostics` — что собралось: сколько ключей/страниц, какие отобраны, целевой
  объём, ошибки Serpstat, и `crawl.failed` (страницы, что не отдали текст).

Коротко покажи пользователю: тип/язык, отобранные ключи, целевой объём, сколько
страниц конкурентов разобрано. Если `crawl.failed` велик (многие сайты под
защитой) — LSI/заголовки будут беднее; можешь догрузить трудные URL через скил
`browse`/`scrape` (gstack) и при желании перезапустить, но это не обязательно.

## Шаг 2. Написать текст

Напиши финальный текст, **СЛЕДУЯ полю `prompt` ДОСЛОВНО** — это и есть свод правил
(объём, плотность, структура, бан-лист, единый HTML-блок с размеченным FAQ и
JSON-LD в конце, мета). Используй `faqSeeds` как источник вопросов для FAQ.
Ключевые правила, которые легко упустить:
- единый HTML-блок: текст → видимый FAQ (`<h2>` блок, `<h3>` вопросы, `<p>`
  ответы) → JSON-LD FAQPage в `<script type="application/ld+json">` в самом конце;
- разрешённые теги только h2/h3/p/ul/ol/li/table/strong + один завершающий script;
- ничего не выдумывай: если факта нет во вводных/`--facts` — пропусти блок, без
  плейсхолдеров;
- **внутренняя перелинковка:** если задан `--site`, collect собирает из карты
  сайта релевантные разделы/бренды (поле `ССЫЛКИ_ПЕРЕЛИНКОВКИ` промпта и
  `diagnostics.internalLinks`). Оберни **2–4** уместных анкора (франшизы, бренды,
  «в каталоге …») в `<a href="…">` из этих кандидатов, ≤1 ссылки на абзац. Анкоры
  в таблице/прозе — норма. `--links "url — тема"` перекрывает, `--no-links` отключает;
- мета (Title/Description) — по разделу 10 документа.

Сохрани написанный текст в файл:

```
Write ~/.claude/skills/seo-text/draft.html  ← сюда твой HTML целиком
```

## Шаг 3. Проверить и переписать до чистоты

```bash
node ~/.claude/skills/seo-text/check.js \
  --text ~/.claude/skills/seo-text/draft.html \
  --brief ~/.claude/skills/seo-text/run.json
```

Прочитай чек-лист. **Каждый `✗` (fail) — исправь** (плотность, объём, бан-лист,
структура, FAQ/мета) и перезапиши `draft.html`, затем прогони проверку снова.
`⚠` — на твоё суждение (часто оправданы). Повтори до отсутствия фейлов или пока
оставшиеся предупреждения осознанно приемлемы.

## Шаг 4. Сохранить и отдать результат

**Сохрани финальный текст в отдельный файл в рабочей папке пользователя и
ОБЯЗАТЕЛЬНО назови его абсолютный путь** — это и есть итог ради которого всё
затевалось, пользователь должен сразу знать, где лежит готовый HTML:

```
Write ./seo-text-<слаг-категории>.html   ← финальный HTML целиком (без строк Title:/Description:)
```

Затем верни пользователю: (1) **путь к сохранённому файлу** (первой строкой),
(2) Title и Description, (3) сам HTML для копирования, (4) короткий отчёт — какие
ключи/структура использованы, итог проверки (✓/⚠/✗ и метрики), какие страницы
конкурентов не удалось разобрать. Рабочие `run.json`/`draft.html` можно убрать, а
итоговый `seo-text-*.html` — ОСТАВЬ.

## Самопроверка перед выдачей (уроки боевых прогонов)

После `check.js`, ДО выдачи, пройдись по списку — он ловит то, что механическая
проверка пропускает:

1. **Курируй ключи.** Set-cover иногда тащит мусор: чужой товар (к «фигуркам Hot
   Toys» — «набор для создания гипсовых фигурок»), чужое/чувствительное гео
   («официальный сайт в россии» для украинского магазина). Выкинь нерелевантное
   из `brief.keywords` (отредактируй `run.json`) и не впихивай в текст — проверяй
   уже по курированному набору. Релевантный лонг-тейл (франшизы, герои, модели)
   бери из СЕМАНТИЧЕСКОГО_ЯДРА словоформами.
2. **Плотность многословного ключа.** Ключ из 3 слов даёт +3 к числителю за
   каждое вхождение — держи ~2 целых вхождения (H1 + лид), остальное словоформами
   и синонимами. H2 пусть содержат части ключа для правила, но без полного
   триграма (не ставь «фигурк_» вплотную к «Hot Toys» в лишних заголовках).
3. **Не пересушивай одним синонимом.** Разбавишь тему только «моделью» — пик
   тошноты переедет на «модель». Чередуй: фигурка / модель / экземпляр /
   статуэтка / герой / персонаж.
4. **Видимый FAQ = JSON-LD дословно.** Правишь видимый ответ — синхронь его копию
   в JSON-LD (тексты должны совпадать).
5. **Нет фактов — нет блоков.** Без `--facts` не пиши «преимущества / доставка /
   гарантия / цена». Цену в FAQ отправляй в карточку товара, без выдуманных чисел.
6. **Краул-провалы — норма.** Маркетплейсы, соцсети, Cloudflare часто не отдают
   текст; работай по разобравшимся + по ядру. Трудные URL при нужде догрузи через
   gstack `browse`.
7. **Стыки элементов.** Заголовок, кончающийся на «…Hot Toys», + следующий,
   начинающийся со слова-ключа, раньше давал фантомное вхождение; check.js это уже
   чинит (счёт по сегментам), но не строй заголовки так нарочно.
8. **Ожидаемые ⚠:** дистанция H1↔лид <300 збп (документ сам требует ключ и в H1, и
   в первом абзаце) и LSI-покрытие <60% на узкой теме после чистки — это норма.

## Заметки

- Скрипты переиспользуют `src/lib/*` репозитория (serpstat, extract, headings,
  pipeline, ecomprompt, textcheck) — один движок, без дублирования логики.
- `node_modules` берётся из корня репозитория (jsdom/readability для краула).
- EN-документы используют слова и плотность %, без рунет-«тошноты»; RU/UK — збп
  и Advego/Text.ru. check.js это учитывает по `lang`.

## Установка (самодостаточный бандл)

Эта папка самодостаточна (свой `lib/` и `package.json`). Установка:

```bash
cp -r <эта папка> ~/.claude/skills/seo-text   # или скопируй вручную
cd ~/.claude/skills/seo-text && npm install     # тянет jsdom + readability
```

После этого `/seo-text` доступен в любом проекте. Пути выше — для установки в `~/.claude/skills/seo-text`; если положишь в другое место, поправь путь в командах.
